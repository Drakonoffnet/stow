# Спецификация требований — `file_transfer`

> Статус: requirements (итог брейншторма). Дизайн и реализация — отдельные шаги.
> Дата: 2026-06-28

## 1. Цель

Десктоп-утилита для **macOS**: принять папку через **drag-and-drop** в GUI-окне,
сжать её в **zip** (с параллельным сжатием), переместить результат в указанное назначение.

## 2. Пользовательский сценарий

1. Пользователь запускает приложение (двойной клик по `.app` или из терминала).
2. Открывается окно с двумя зонами:
   - **«Источник»** — перетащить папку(и) для упаковки;
   - **«Назначение»** — перетащить папку, куда положить готовый архив.
3. Нажимает «Старт» → видит прогресс по каждому архиву.
4. По завершении — статус (успех/ошибка) и путь к результату.

## 3. Стек и структура (в рамках текущего крейта)

```
file_transfer/
├── Cargo.toml          # +зависимости, [lib] + [[bin]]
└── src/
    ├── lib.rs          # ЯДРО: архивация + перемещение (без GUI, тестируемо)
    │   ├── archive.rs  #   zip + параллельное сжатие (rayon)
    │   ├── transfer.rs #   бэкенды назначения (trait Destination)
    │   └── job.rs      #   модель задачи, прогресс, статусы
    └── main.rs         # GUI-слой на egui/eframe (тонкая обёртка над lib)
```

- **Ядро (`lib`)** не зависит от GUI → тестируется отдельно, при желании можно прикрутить CLI.
- **GUI (`bin`)** вызывает ядро, рисует drop-зоны и прогресс.
- Ключевые крейты: `eframe`/`egui` (GUI + drag-drop), `zip` (формат),
  `rayon` (параллелизм), `sha2` (checksum).
  Этап 2: `ssh2`/`openssh`, `aws-sdk-s3`.

## 4. Функциональные требования

| ID  | Требование                                                            | Этап   |
| --- | -------------------------------------------------------------------- | ------ |
| F1  | GUI-окно, drop-зона **«Источник»** — приём одной/нескольких папок     | MVP    |
| F2  | GUI drop-зона **«Назначение»** — локальная папка                      | MVP    |
| F3  | Каждая папка → отдельный `.zip`                                       | MVP    |
| F4  | **Параллельное** сжатие файлов внутри архива (`rayon`), настройка jobs | MVP    |
| F5  | Очередь: несколько архивов обрабатываются одновременно                | MVP    |
| F6  | Прогресс-бар и статус по каждой задаче                                | MVP    |
| F7  | Атомарность: tmp-файл → перемещение только после успешной упаковки     | MVP    |
| F8  | Исходники сохраняются; опция «удалить после успеха»                   | MVP    |
| F9  | sha256 готового архива (опционально)                                  | MVP    |
| F10 | Лог в окне + файл                                                     | MVP    |
| F11 | Бэкенд назначения: **сетевой диск (SMB/NFS как путь)**                | MVP \* |
| F12 | Бэкенд: **SSH/SCP** (`trait Destination`)                            | Этап 2 |
| F13 | Бэкенд: **облако S3**                                                | Этап 2 |

\* сетевой диск работает как обычный путь — попадает в MVP автоматически.

## 5. Нефункциональные требования

- macOS (Apple Silicon + Intel), сборка в `.app`.
- Утилизация всех ядер CPU при сжатии.
- Ядро покрыто unit-тестами (архивация/перемещение независимо от GUI).
- Дружелюбие к запуску из терминала (коды возврата).

## 6. Открытые вопросы (решить перед дизайном)

1. **Креды SSH/облака** — где хранить (конфиг / поля GUI / Keychain)?
   Предложение: вынести в этап 2.
2. **Уровень сжатия** — фиксированный (6) или ползунок в GUI?
3. **Имя архива** — `<имя_папки>_<дата>.zip`? Поведение при коллизии (перезапись/суффикс)?
4. **Назначение для SSH/облака** — drop-зона им не подходит, нужен отдельный селектор настроек.

## 7. Принятые решения (контекст брейншторма)

- Параллельность: **один архив сжимается многопоточно** (в zip — параллельное сжатие
  отдельных файлов внутри архива).
- Язык: **Rust** (единый стек с текущим проектом).
- Формат: **zip**.
- Назначения: локальная папка, сетевой диск, SSH/SCP, облако (поэтапно).
- Интерфейс: **GUI-окно с drag-and-drop**.
- ОС: **macOS**.
- Размещение: **часть текущего крейта**, с разделением **lib-ядро + GUI-слой**.
